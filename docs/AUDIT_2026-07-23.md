# PUM audit — 2026-07-23

Scope: repository `Supersynergy/pum`, installed `pum 0.1.2`, and one isolated
live `scan`/`check`. This is an implementation audit, not a claim that every
package manager can expose a non-mutating latest-version query.

## Verified baseline

- Rust CLI, 12 adapters: brew, npm, pnpm, bun, uv, pipx, cargo, rustup, gem,
  go, mise, and GitHub CLI extensions.
- Commands: `scan`, `check`, `report`, guarded `update`, guarded manager
  `self`, `doctor`, project dependency checks, and OSV audit.
- Safety boundary holds: no OS updater; `scan` and `check` do not upgrade.
- Existing persistence was one mutable SQLite `tools` table plus a JSON mirror.
- Isolated live run: 798 installed packages and 35 known upgrades: Homebrew 26,
  npm 7, Cargo 1, RubyGems 1.
- Published release was `v0.1.2` on 2026-07-02, with five cargo-dist target
  archives and checksums.

## What was missing

| Gap | Consequence | v0.2 response |
|---|---|---|
| SQLite-only current-state table | No durable answer to “what was latest yesterday?” | DuckDB inventory plus `refresh_runs` and `version_observations` |
| `scan` JSON mirror used raw scan rows | A fresh JSON mirror could omit prior latest/status values | Rewrite it from the persisted inventory |
| No single source-refresh command | Agent had to coordinate scan, check, report itself | `pum refresh --json` performs a sequential read-only snapshot |
| No freshness/status contract | Agent could not know whether data was stale | `pum status --json` returns age, candidates, source coverage, and DB path |
| No daily check | Latest metadata depended on manual use | Opt-in macOS `pum schedule --install` invokes only `pum refresh --json` |
| No agent-native package freshness interface | Agents had to shell out and parse CLI output | `pum mcp` serves a local stdio MCP with no package-mutation tool |
| Partial native source coverage | Empty output could be mistaken for “current” | Coverage is explicit: `candidate_versions` vs `update_only` |

## Source truth, per adapter

`candidate_versions` means PUM runs the manager's non-mutating query and saves
actual update candidates: Homebrew, npm, pnpm, Cargo (when
`cargo-install-update` exists), RubyGems, mise, and rustup.

`update_only` means the manager has a guarded updater but PUM does not yet have
a reliable non-mutating, package-level latest query: bun, uv tools, pipx, Go
binaries, and GitHub CLI extensions. Their inventory remains useful, but PUM
must not label them current. Dedicated source resolvers are the next feature,
not a release claim.

## Storage decision

Use embedded DuckDB, not DuckLake. DuckLake adds a catalog/object-store layer
for shared lakehouse data; PUM is a local, single-user inventory with a daily
append-only history. DuckDB enables direct SQL, history aggregation, and future
Parquet export while keeping one local file. `rusqlite` remains temporarily
only to import a pre-0.2 `inventory.db` into the new default
`inventory.duckdb`; normal runtime persistence is DuckDB.

## Remaining requirements before broad release

1. Implement source resolvers for each `update_only` adapter, or preserve that
   explicit label indefinitely.
2. Add signed PUM self-update metadata before allowing the binary itself to
   update automatically. A GitHub “latest” tag alone is not enough.
3. Gate tag releases on tests, clippy, cargo-deny, release build, and a clean
   `pum refresh --json` smoke on every native target.
4. Verify published archive hashes and installer behavior from the immutable
   release asset, not only the GitHub `latest` alias.
5. Before a public tag, pin every GitHub Action in CI/release workflows to a
   reviewed full commit SHA and replace the existing `curl | sh` cargo-dist
   bootstrap with a checksum/signature-verified download. The security review
   currently flags these workflow-level supply-chain gaps.

## 2026-07-23 follow-up

The tree now pins every workflow action to a reviewed full commit SHA, uses
Cargo's checksum-verified registry install for `cargo-dist`, and refuses a
container Rust bootstrap unless `rustup` already exists. The release token is
read-only outside the publishing job. These changes still require a successful
remote CI/release run before they become an online release claim.
